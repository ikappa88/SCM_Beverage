# セキュリティ注意事項

## 本番環境移行前に必ず対応すること

### 高優先度

1. **SECRET_KEY の変更**
   - `backend/.env` の `SECRET_KEY` を十分な長さのランダム文字列に変更する
   - 生成例: `openssl rand -hex 32`

2. **DBパスワードの変更**
   - `backend/.env` の `DATABASE_URL` のパスワードを変更する
   - `docker-compose.env` に本番用パスワードを設定する

3. **CORS設定の変更**
   - `backend/app/main.py` の `allow_origins` を本番URLに変更する
   - 環境変数化を推奨

4. **HTTPS の有効化**
   - 本番環境では必ずHTTPSを使用する
   - リバースプロキシ（Nginx等）でSSL終端を行う

5. **APIドキュメントの保護**
   - 本番環境では `/docs` と `/redoc` を無効化する
   - `app = FastAPI(docs_url=None, redoc_url=None)` で無効化

### 中優先度

6. **トークン保管方式の変更**
   - 現在: localStorage（XSSリスクあり）
   - 推奨: httpOnly Cookie + CSRF対策

7. **ログイン試行制限**
   - ブルートフォース対策としてレート制限を実装する
   - slowapi等のライブラリを検討

8. **管理者の自己無効化ガード**
   - 最後の管理者アカウントを無効化できないよう制御を追加する

### 低優先度（運用上の改善）

9. **Pydanticバリデーションの強化**
   - 文字列の最大長・最小長・パターン制約を追加する

10. **監査ログの長期保管**
    - ログのアーカイブ・ローテーション方針を定める

## 既知の制限事項（開発版）

- `localStorage` によるトークン保管: XSS攻撃時にトークンが流出するリスクがある
- ログイン試行制限なし: ブルートフォース攻撃への対策が不十分
- `/docs` エンドポイント: 開発環境では有効のまま
