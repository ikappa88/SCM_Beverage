# シミュレーション機能 要件定義

> バージョン: v2.0
> 作成日: 2026-04-06（v1.0: 2026-04-05）
> 変更概要: シミュレーション視点を地域TC担当目線に刷新。上流（工場・広域DC）と下流（小売）の自動動作を追加し、荷動き計画ビューを新設。

---

## 1. 背景・目的

### 1.1 背景

本システムは飲料サプライチェーンの地域TC（Transfer Center）担当者向けシミュレーターである。TC担当者は広域DCから商品を受け取り、配下のスーパー・コンビニ等の小売店へ配送する中継拠点を運営する。

**上流・下流の動きはシステムが自動的に模擬**し、TC担当者はその中で実際の業務判断（発注・在庫管理・アラート対応）を行うことでスキルを習得する。

### 1.2 サプライチェーン全体像

```
工場（Factory）
  │ 生産スケジュールに基づき自動生産・出荷
  ▼
広域DC（Wide-area Distribution Center）
  │ 工場からの入荷を自動受領。TC発注に応答して出荷
  ▼
┌─────────────────────────────────────┐
│  地域TC（Regional Transfer Center）  │  ← TC担当オペレーターが操作するレイヤー
│  在庫管理・発注・アラート対応         │
└─────────────────────────────────────┘
  │ 需要モデルに基づき自動消費
  ▼
小売店（スーパー・コンビニ等 / Retail）
  │ 消費者需要
  ▼
消費者
```

### 1.3 レイヤー別 自動処理 vs. 手動操作

| レイヤー | 動作主体 | 内容 |
|---------|---------|------|
| 工場 | **システム自動** | 生産スケジュール実行、広域DCへ自動出荷 |
| 広域DC | **システム自動** | 工場在庫受領、TC発注への自動出荷応答 |
| 地域TC | **TC担当オペレーター** | 広域DCへの発注、在庫管理、アラート対応、配送計画確認 |
| 小売店 | **システム自動** | 需要モデルによる自動注文・TC在庫消費 |

### 1.4 目的

- TC担当者が、上流・下流の動きに左右される中で実際の業務判断を体験する
- 発注タイミングの遅れ・在庫切れ・配送遅延などの問題を体験的に学ぶ
- 先々の荷動き計画を読み解き、先手を打つ意思決定の練習をする

### 1.5 前提

- 本 DB がシミュレーション環境。フラグ管理・DB 分離は不要
- 時刻は仮想時刻（Virtual Clock）で管理し、実時刻とは独立して動作する
- `location_type` で工場・広域DC・地域TC・小売店を区別する（既存 Location モデルに追加）

## 2. 仮想時刻（Virtual Clock）

### 2.1 概念

システム内にグローバルな「現在の仮想時刻」を保持する。全ての業務ロジックはこの仮想時刻を基準に動作する。

### 2.2 時間進行の単位

| 操作 | 進行量 | 仮想時刻の変化例 |
|------|--------|-----------------|
| 「半日進める」ボタン 1クリック | +12時間 | 4/5 AM 09:00 → 4/5 PM 21:00 → 4/6 AM 09:00 |

- AM セッション: 00:00〜11:59（代表時刻 09:00）
- PM セッション: 12:00〜23:59（代表時刻 21:00）
- クリック毎に 12 時間加算、AM/PM が交互に切り替わる

### 2.3 永続化・表示

- `simulation_clock` テーブルに単一レコードとして保存（複数ブラウザ間共有）
- ヘッダーに常時表示（例: `2026-04-05 (PM)`、AM/PM をバッジで色分け）
- 管理者のみ「リセット」操作で初期日時に戻せる

## 3. 時間進行時の自動処理（advance）

「半日進める」ボタン押下時、以下の順序で処理する。

### 3.1 処理順序

```
1. 仮想時刻の更新
2. 工場の生産・出荷（上流自動）
3. 広域DC入荷処理（上流自動）
4. 広域DC → TC 配送ステータス進行（上流自動）
5. TC → 小売 消費（下流自動）
6. TC 発注ステータスの進行（TC担当の発注に対するシステム応答）
7. アラート評価・発火
8. イベントログ記録
```

---

### 3.2 工場の生産・出荷（上流自動）

#### 3.2.1 生産モデル

工場は「生産スケジュール」パラメータに基づき、半日ごとに在庫（広域DC向け）を生産する。

```
生産量（半日分）= 基本生産量（日量）
               × 0.5（半日係数）
               × 季節係数（月別）
               × ランダムノイズ（±10%）
```

- 基本生産量は SKU ごとにパラメータ管理
- **ランダム事象**（ライン停止・原材料不足）: 設定確率で生産量ゼロになる半日が発生し、サプライ不足を模擬する

#### 3.2.2 工場 → 広域DC への自動出荷

- 生産完了分は即日、広域DC向け配送レコードを自動生成する
- 工場→広域DC のリードタイムはパラメータ管理（デフォルト 2 半日）
- 配送到着時に広域DC在庫へ自動加算

---

### 3.3 広域DC 入荷処理（上流自動）

- 工場からの配送（§3.2.2）が到着した場合、広域DC の在庫を加算する
- 広域DC在庫が一定量を下回った場合、`wide_dc_low_stock` アラートを生成する（TC担当への情報提供）

---

### 3.4 広域DC → TC 配送ステータス進行（上流自動）

TC担当が広域DCへ発注した注文は、以下のステータスを自動遷移する。

| 現ステータス | 遷移条件 | 次ステータス |
|------------|---------|------------|
| `confirmed` | 次の AM 到来 | `preparing` |
| `preparing` | 1 半日経過 | `shipped`（配送レコード自動生成） |
| `shipped` / `in_transit` | リードタイム経過 | `delivered`（TC 在庫に自動加算） |

**リードタイム（広域DC → TC）:**

```
実効リードタイム = 基本リードタイム + ランダムゆらぎ（-1〜+2 半日）
```

- 週末を跨ぐ場合は +1〜2 半日を加算
- **広域DC在庫が不足**している場合: `shipped` への遷移を保留し `wide_dc_shortage` イベントを記録する

---

### 3.5 TC → 小売 消費（下流自動）

小売需要モデルに基づき、TC在庫から半日分の需要を自動消費する。

#### 3.5.1 需要計算式

```
需要量（半日分）= 基本需要量（日量・SKU別）
               × 0.5（半日係数）
               × 曜日係数
               × 季節係数（月別）
               × 気温需要係数（AM/PM別）
               × ランダムノイズ（±20%）
```

#### 3.5.2 需要係数テーブル（デフォルト値）

**曜日係数:**

| 月 | 火 | 水 | 木 | 金 | 土 | 日 |
|----|----|----|----|----|----|-----|
| 0.9 | 0.9 | 1.0 | 1.0 | 1.1 | 1.3 | 1.2 |

**季節係数（月別）:**

| 1月 | 2月 | 3月 | 4月 | 5月 | 6月 | 7月 | 8月 | 9月 | 10月 | 11月 | 12月 |
|0.7|0.7|0.9|1.0|1.1|1.2|1.5|1.6|1.2|1.0|0.8|0.8|

**気温需要係数（月別 AM/PM、気象庁東京月別平均気温参照）:**

| 月 | AM | PM |
|----|----|-----|
| 1〜2月 | 0.8 | 0.8 |
| 3〜4月 | 0.9 | 1.0 |
| 5〜6月 | 1.0 | 1.2 |
| 7〜9月 | 1.1 | 1.5 |
| 10〜11月 | 0.9 | 1.0 |
| 12月 | 0.8 | 0.8 |

> 参照: 気象庁 過去の気象データ（東京・月別平均気温）

#### 3.5.3 在庫不足の場合

- 需要量が TC 在庫を超える場合は在庫を 0 止め（マイナスにしない）
- 欠品量をイベントログに記録し、`stockout` アラートを発火する

---

### 3.6 TC 発注ステータスの進行

- §3.4 に従い、広域DCへの発注は時間経過で自動遷移する
- `draft` のまま放置した発注は自動遷移しない（TC担当の操作が必要）

---

### 3.7 アラートの評価・発火

時間進行後に以下の条件を評価し、未発火のアラートを生成する。

| アラート種別 | 発火条件 | 対象レイヤー |
|------------|---------|------------|
| 在庫切れ | TC在庫 = 0 | TC |
| 在庫低下 | TC在庫 ≤ safety_stock | TC |
| ATP 不足 | ATP ≤ 0 | TC |
| 賞味期限超過 | ロット賞味期限 ≤ 仮想現在日 | TC |
| 賞味期限間近 | ロット賞味期限 ≤ 仮想現在日 + 警告日数 | TC |
| 配送遅延 | 予定配達日 ≤ 仮想現在日 かつ未着 | TC |
| 広域DC在庫低下 | 広域DC在庫 ≤ wide_dc_safety_stock | 広域DC（情報提供） |
| 広域DC欠品 | TC発注に対し広域DCが在庫不足で出荷保留 | 広域DC（緊急） |
| 工場供給停止 | ランダム事象で工場生産ゼロが発生 | 工場（情報提供） |

- 同一条件の重複アラートは発火しない
- スヌーズ中のアラートは条件悪化時のみ再発火する

## 4. 荷動き計画ビュー（新設）

TC担当者が意思決定するための「先々の荷動き可視化」機能。

### 4.1 概要

現在の仮想時刻から **N 日先**（デフォルト 14 日）までの以下を一覧表示する。

| 情報 | 内容 |
|------|------|
| 確定入荷予定 | 広域DCから到着予定の配送（ステータス・数量・予定日） |
| 需要予測（出荷予定） | 需要モデルによる各日の推定消費量 |
| 在庫推移グラフ | 現在庫 + 入荷予定 − 需要予測 の日別積み上げ |
| ATP（利用可能在庫） | 既存算式に基づくリアルタイム計算値 |
| 欠品予測日 | 在庫推移が 0 以下になる最初の日付 |
| 広域DC在庫状況 | 広域DCの現在庫（TC担当が参照可能な範囲） |

### 4.2 表示形式

- **カレンダー形式のタイムライン**（横軸: 日付、縦軸: SKU）
  - 各セルに「入荷予定数量」「需要予測数量」を色分けで表示
  - 欠品が予測される日付をハイライト（赤背景）
- **在庫推移グラフ**（折れ線グラフ）
  - 現在庫からの推移を日単位で表示
  - safety_stock ラインを水平線で表示
- **広域DCステータスサマリー**
  - 広域DC在庫水準（十分 / 注意 / 不足）をバッジで表示

### 4.3 更新タイミング

- 「半日進める」完了後に自動更新
- 手動リロードボタンも提供

### 4.4 API エンドポイント（追加）

| メソッド | パス | 説明 |
|---------|------|------|
| `GET` | `/api/simulation/movement-plan` | 荷動き計画データ取得（SKU・拠点・日数を指定） |

## 5. TC 担当者の操作フロー（シミュレーション中の業務）

### 5.1 標準的な1日の流れ

```
AM ボタン押下
  → 工場生産・広域DC入荷・TC消費が自動実行
  → アラートを確認
  → 在庫低下 or ATP不足 が見えた場合、広域DCへ発注（confirmed）
  → 荷動き計画ビューで今後の在庫推移を確認

PM ボタン押下
  → 引き続き消費・配送進行が自動実行
  → 確定入荷の状況を確認
  → 翌日以降の発注量を計画
```

### 5.2 判断ポイント（学習目標）

| 状況 | TC担当が取るべきアクション |
|------|----------------------|
| 在庫が safety_stock を下回った | 広域DCへ補充発注（適切な数量・タイミング） |
| 広域DC在庫低下アラート | 早めに多めに発注 or 代替品検討 |
| 広域DC欠品アラート（出荷保留） | 他 TC からの融通・配送計画の見直し |
| 賞味期限間近 | ロット優先出荷の指示・値引き販売の判断 |
| 配送遅延 | 得意先への連絡・代替便の手配 |
| 工場供給停止 | 先行在庫を確保する早期発注 |

## 7. イベントログ

半日進める操作ごとに発生した全イベントを保存し、UI から確認できる。

### 7.1 イベント種別一覧

| event_type | 内容 | 発生レイヤー |
|------------|------|------------|
| `factory_produced` | 工場が生産完了・広域DCへ出荷 | 工場 |
| `factory_line_stopped` | ランダム事象でライン停止 | 工場 |
| `wide_dc_received` | 広域DCが工場から入荷 | 広域DC |
| `wide_dc_shipped` | 広域DCがTC向けに出荷 | 広域DC |
| `wide_dc_shortage` | TC発注に対し広域DCが在庫不足で保留 | 広域DC |
| `delivery_status_changed` | 配送ステータスが遷移 | 配送 |
| `delivery_arrived` | TC に配送到着・在庫加算 | TC |
| `inventory_consumed` | 小売需要でTC在庫が消費 | TC |
| `stockout` | TC在庫が欠品 | TC |
| `order_status_changed` | TC発注のステータスが遷移 | TC |
| `alert_fired` | アラート発火 | 各レイヤー |

### 7.2 UI 表示

- サイドパネルで直近 N 件を時系列表示
- レイヤー（工場 / 広域DC / TC）ごとにアイコン・色分け
- 日本語サマリー表示

## 11. 非機能要件

| 項目 | 要件 |
|------|------|

| ログ保持期間 | イベントログは最低 2000 件保持 |
