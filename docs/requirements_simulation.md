# シミュレーション機能 要件定義

> バージョン: v1.0
> 作成日: 2026-04-05
> 対象システム: SCM Beverage（本システム全体がシミュレーション環境として動作する）

---

## 1. 背景・目的

### 1.1 背景

本システムは飲料サプライチェーンの管理シミュレーターとして設計されている。実際の業務フローを仮想的に体験することで、オペレーターが発注・在庫管理・配送管理の意思決定を練習できる環境を提供する。

### 1.2 目的

- オペレーターが安全な環境で業務判断のトレーニングを行う
- 在庫切れ・過剰在庫・配送遅延などの問題発生を体験的に理解する
- アラート対応や ATP（利用可能在庫）の動的変化を実感として学ぶ

### 1.3 前提

- 本システム上のデータは全てシミュレーションデータであり、本番環境とは別である
- シミュレーションフラグや DB 分離は行わない（本 DB が唯一の真実）
- 時刻は「仮想時刻（Virtual Clock）」で管理し、実際のシステム時刻とは独立して動作する

---

## 2. 仮想時刻（Virtual Clock）

### 2.1 概念

システム内に「現在の仮想時刻」をグローバルな状態として保持する。全ての業務ロジック（在庫消費・配送ステータス変化・アラート判定）はこの仮想時刻を基準に動作する。

### 2.2 時間進行の単位

| 操作 | 進行量 | 仮想時刻の変化例 |
|------|--------|-----------------|
| 「半日進める」ボタン 1クリック | +12時間 | 4/5 AM 09:00 → 4/5 PM 21:00 → 4/6 AM 09:00 |

- AM セッション: 仮想日付 00:00 〜 11:59（代表時刻 09:00）
- PM セッション: 仮想日付 12:00 〜 23:59（代表時刻 21:00）
- クリック毎に 12 時間加算し、AM/PM が交互に切り替わる

### 2.3 仮想時刻の永続化

- 仮想時刻はサーバー側 DB に保存する（`simulation_clock` テーブル）
- 複数のブラウザセッション間で共有される（単一の仮想時刻が存在する）
- 「リセット」操作により、管理者が設定した初期日時に戻す

### 2.4 仮想時刻の表示

- ヘッダーに常時表示する（例：「仮想時刻: 2026-04-05 (PM)」）
- AM/PM をバッジで色分けして視認性を高める

## 3. 時間進行時に発生するイベント

「半日進める」ボタン押下時、以下のイベントが順序通りに処理される。

### 3.1 処理順序

```
1. 仮想時刻の更新
2. 配送ステータスの自動進行
3. 在庫消費（需要モデルに基づく出荷）
4. 注文ステータスの自動進行
5. ATP の再計算
6. アラートの評価・発火
7. イベントログの記録
```

### 3.2 配送ステータスの自動進行

配送の各ステータスはリードタイム（半日単位）の経過で自動遷移する。

| 現ステータス | 遷移条件 | 次ステータス |
|------------|---------|------------|
| `confirmed` | 半日経過 | `preparing`（出荷準備中） |
| `preparing` | 半日経過 | `shipped`（出荷済み） |
| `shipped` | リードタイム経過 | `in_transit`（配送中） |
| `in_transit` | リードタイム経過 | `delivered`（配達完了） |

**リードタイムの計算式（半日単位）:**

```
実効リードタイム = 基本リードタイム × 曜日係数 + ランダムゆらぎ
ランダムゆらぎ    = -1 〜 +2 半日（一様分布）
```

- 基本リードタイムはサプライヤーごとにパラメータ管理する（単位: 半日数）
- 曜日係数: 週末を跨ぐ場合は +1〜2 半日を加算（輸送会社の休日考慮）
- `delivered` 到達時: 受取拠点の在庫が自動加算される

### 3.3 在庫消費（需要モデルに基づく出荷）

半日ごとに各拠点・SKU の需要量を計算し、在庫から減算する。

#### 3.3.1 需要計算式

```
需要量（半日分）= 基本需要量（日量）
                  × 0.5（半日係数）
                  × 曜日係数
                  × 季節係数（月別）
                  × 気温需要係数（AM/PM別）
                  × ランダムノイズ
```

#### 3.3.2 需要パラメータ

| パラメータ | 内容 | 根拠 |
|-----------|------|------|
| **基本需要量（日量）** | 拠点・SKU ごとに設定する数量 | 初期値はシステム管理者が投入 |
| **曜日係数** | 月〜日の消費量倍率 | 下表参照 |
| **季節係数（月別）** | 1〜12 月の消費量倍率 | 下表参照 |
| **気温需要係数** | AM（涼しい）/ PM（暑い）の係数 | 月別平均気温（気象庁公表値）を参照 |
| **ランダムノイズ** | ±20% の一様乱数 | 日常的な需要変動を再現 |

**曜日係数（デフォルト値）:**

| 月 | 火 | 水 | 木 | 金 | 土 | 日 |
|----|----|----|----|----|----|-----|
| 0.9 | 0.9 | 1.0 | 1.0 | 1.1 | 1.3 | 1.2 |

**季節係数（月別デフォルト値）:**

| 1月 | 2月 | 3月 | 4月 | 5月 | 6月 | 7月 | 8月 | 9月 | 10月 | 11月 | 12月 |
|-----|-----|-----|-----|-----|-----|-----|-----|-----|------|------|------|
| 0.7 | 0.7 | 0.9 | 1.0 | 1.1 | 1.2 | 1.5 | 1.6 | 1.2 | 1.0  | 0.8  | 0.8  |

**気温需要係数（月別 AM/PM 別、気象庁東京月別平均気温を基に設定）:**

| 月 | AM係数 | PM係数 | 備考 |
|----|--------|--------|------|
| 1〜2月 | 0.8 | 0.8 | 冬季、気温低く清涼飲料消費少 |
| 3〜4月 | 0.9 | 1.0 | 春、昼間の消費が増加 |
| 5〜6月 | 1.0 | 1.2 | 初夏、PM の消費が顕著に増加 |
| 7〜9月 | 1.1 | 1.5 | 夏季ピーク、PM は最大消費 |
| 10〜11月 | 0.9 | 1.0 | 秋 |
| 12月 | 0.8 | 0.8 | 冬 |

> 参照: [気象庁 過去の気象データ](https://www.data.jma.go.jp/stats/etrn/index.php)（東京・月別平均気温）

#### 3.3.3 在庫がゼロの場合

- 需要量が在庫を超える場合は在庫は 0 止まり（マイナスにしない）
- 欠品イベントとしてイベントログに記録する
- 欠品アラートを発火する

### 3.4 注文ステータスの自動進行

オペレーターが作成した発注は、時間経過でステータスが変化する。

| 現ステータス | 遷移条件 | 次ステータス |
|------------|---------|------------|
| `draft` | ─（自動遷移なし） | ─ |
| `confirmed` | AM 押下時に1回 | `preparing` |
| `preparing` | 次の半日経過 | `shipped` → 配送レコード生成 |

- `draft` のまま放置した注文は時間が経過しても変化しない（要オペレーター操作）
- `confirmed` 移行には、現在の要件定義（requirements_v3.md）の承認フローを踏む

### 3.5 ATP の再計算

在庫消費・配送完了・注文ステータス変化後に自動で ATP を再計算する。

```
ATP = 現在庫数
    + Σ confirmed/in_transit の入荷予定数量
    - Σ 確定済み出荷予定数量
```

- ATP がゼロ以下になった SKU はアラートを発火する

### 3.6 アラートの評価・発火

時間進行後に以下の条件を評価し、未発火のアラートを生成する。

| アラート種別 | 発火条件 |
|------------|---------|
| 在庫切れ | 在庫 = 0（欠品発生） |
| 在庫低下 | 在庫 ≤ reorder point |
| ATP 不足 | ATP ≤ 0 |
| 賞味期限超過 | ロットの賞味期限 ≤ 仮想現在日 |
| 賞味期限間近 | ロットの賞味期限 ≤ 仮想現在日 + 警告日数 |
| 配送遅延 | 予定配達日 ≤ 仮想現在日 かつ `delivered` でない |

- 同一条件の重複アラートは発火しない（既存アラートが `open` の場合はスキップ）
- スヌーズ中のアラートは条件が悪化した場合のみ再発火する

## 4. パラメータ管理

### 4.1 パラメータ一覧

| カテゴリ | パラメータ名 | 管理単位 | 変更権限 |
|---------|------------|---------|---------|
| 需要 | 基本需要量（日量） | 拠点 × SKU | 管理者 |
| 需要 | 曜日係数テーブル | システム全体 | 管理者 |
| 需要 | 季節係数テーブル | システム全体 | 管理者 |
| 需要 | 気温需要係数テーブル | AM/PM × 月 | 管理者 |
| 需要 | ランダムノイズ範囲（%） | システム全体 | 管理者 |
| 配送 | 基本リードタイム | サプライヤー | 管理者 |
| 配送 | リードタイムゆらぎ範囲（半日） | システム全体 | 管理者 |
| 在庫 | reorder point | 拠点 × SKU | 管理者 |
| 在庫 | 賞味期限警告日数 | SKU | 管理者 |
| 時計 | 仮想時刻の初期値 | システム全体 | 管理者 |

### 4.2 パラメータの保存場所

- DB の `simulation_parameters` テーブルで管理する（Key-Value 形式 + カテゴリ列）
- 管理者 UI からテーブル形式で閲覧・編集できる
- 変更履歴はログに残す（いつ誰が変えたか）

## 5. イベントログ

### 5.1 概要

「半日進める」操作ごとに発生した全イベントをログとして保存し、UI から確認できる。

### 5.2 ログの内容

```json
{
  "virtual_time": "2026-04-05T21:00:00",
  "half_day": "PM",
  "events": [
    {
      "type": "inventory_consumed",
      "location": "東京DC",
      "sku": "コーラ 500mL",
      "quantity": 48,
      "remaining_stock": 312
    },
    {
      "type": "delivery_status_changed",
      "delivery_id": "DEL-0042",
      "from": "in_transit",
      "to": "delivered"
    },
    {
      "type": "alert_fired",
      "alert_type": "low_stock",
      "location": "大阪DC",
      "sku": "緑茶 350mL"
    },
    {
      "type": "stockout",
      "location": "名古屋DC",
      "sku": "スポーツドリンク 500mL",
      "shortage": 12
    }
  ]
}
```

### 5.3 UI での表示

- サイドパネル（またはモーダル）で直近 N 件のイベントを時系列表示する
- イベント種別ごとにアイコン・色分けを行う
- 「何が起きたか」を日本語でサマリー表示する（例: 「東京DC: コーラ 48本出荷消費」）

## 6. UI 要件

### 6.1 シミュレーションコントロールパネル

画面上部（ヘッダーまたはサブヘッダー）に常時表示する。

| 要素 | 内容 |
|------|------|
| 仮想時刻表示 | 「2026-04-05 (PM)」形式、AM/PM をバッジ表示 |
| 「半日進める」ボタン | 押下で +12時間。処理中はローディング状態 |
| 「イベントログ」ボタン | 直近の変化ログをパネル表示 |
| 「リセット」ボタン | 管理者のみ表示。確認ダイアログ付き |

### 6.2 処理中の UX

- ボタン押下中（API 処理中）はボタンを無効化しスピナーを表示する
- 処理完了後、画面上の在庫数・ステータスを自動更新する（ページリロード不要）
- 処理結果のサマリーをトースト通知で表示する（例:「PM に進みました。イベント 7件発生。」）

### 6.3 アクセス制御

| 操作 | オペレーター | 管理者 |
|------|------------|--------|
| 半日進める | ○ | ○ |
| イベントログ閲覧 | ○ | ○ |
| パラメータ編集 | ✗ | ○ |
| リセット | ✗ | ○ |

## 7. API 設計（概要）

### 7.1 エンドポイント

| メソッド | パス | 説明 |
|---------|------|------|
| `GET` | `/api/simulation/clock` | 現在の仮想時刻を取得 |
| `POST` | `/api/simulation/advance` | 半日進める（イベントを処理して返す） |
| `POST` | `/api/simulation/reset` | 初期時刻にリセット（管理者のみ） |
| `GET` | `/api/simulation/events` | イベントログ一覧取得 |
| `GET` | `/api/simulation/parameters` | パラメータ一覧取得 |
| `PATCH` | `/api/simulation/parameters` | パラメータ更新（管理者のみ） |

### 7.2 `/api/simulation/advance` レスポンス例

```json
{
  "previous_virtual_time": "2026-04-05T09:00:00",
  "new_virtual_time": "2026-04-05T21:00:00",
  "half_day": "PM",
  "event_count": 7,
  "events": [ ... ],
  "alerts_fired": 2,
  "stockouts": ["名古屋DC / スポーツドリンク 500mL"]
}
```

## 8. DB スキーマ（追加テーブル）

### 8.1 `simulation_clock`

| カラム | 型 | 説明 |
|-------|-----|------|
| `id` | INTEGER PK | 常に id=1 の単一レコード |
| `virtual_time` | TIMESTAMP | 現在の仮想時刻 |
| `initial_time` | TIMESTAMP | リセット先の初期時刻 |
| `updated_at` | TIMESTAMP | 最終更新日時 |

### 8.2 `simulation_events`

| カラム | 型 | 説明 |
|-------|-----|------|
| `id` | INTEGER PK | |
| `virtual_time` | TIMESTAMP | イベント発生時点の仮想時刻 |
| `half_day` | VARCHAR(2) | `AM` or `PM` |
| `event_type` | VARCHAR(50) | イベント種別 |
| `payload` | JSONB | イベント詳細（拠点・SKU・数量など） |
| `created_at` | TIMESTAMP | 実時間での記録日時 |

### 8.3 `simulation_parameters`

| カラム | 型 | 説明 |
|-------|-----|------|
| `id` | INTEGER PK | |
| `category` | VARCHAR(50) | `demand` / `delivery` / `stock` / `clock` |
| `key` | VARCHAR(100) | パラメータ名（例: `weekday_coefficient`） |
| `value` | JSONB | 値（数値・配列・オブジェクト） |
| `description` | TEXT | 説明文 |
| `updated_by` | INTEGER FK | 最終更新ユーザー |
| `updated_at` | TIMESTAMP | 最終更新日時 |

## 9. 非機能要件

| 項目 | 要件 |
|------|------|
| レスポンスタイム | 「半日進める」処理は 3秒以内に完了すること |
| 同時実行制御 | 複数ユーザーが同時に「半日進める」を押した場合、2回目はエラーまたはロックで弾く |
| 再現性 | ランダムノイズにシード値オプションを設け、デバッグ時に再現可能にする（管理者設定） |
| ログ保持期間 | イベントログは最低 1000件（仮想時刻のセッション）保持する |
