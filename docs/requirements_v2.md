## 1. 本文書の位置づけ

### 1.1 目的

本文書は、初版要件定義書（BevChain-REQ-001、2026年3月17日）およびギャップ分析・設計書（BevChain-DESIGN-001）をベースとして、**現行実装状況を正確に反映した要件定義書**として改訂したものである。

### 1.2 改訂の背景

初版（v1）作成後、以下の機能が実装済みとなった：

- 発注管理（OP-04）：発注モデル・API・UI完成
- 配送管理（OP-05）：配送記録モデル・API・UI完成

これらの実装完了を受け、**業務フロー全体を通じたシステム評価**を行い、新たな改修課題を洗い出した。

## 2. システム全体像と現行実装状況

### 2.1 対象業務領域

飲料メーカーの多段階サプライチェーンにおける物流管理。

```
工場（Factory）
  ↓ 製品出荷
広域物流センター（DC: Distribution Center）
  ↓ 地域配送
地域配送センター（TC: Transfer Center）
  ↓ 小口配送
小売チャネル（Retail）
```

各段階において在庫・輸送・需要の三要素を一元管理し、欠品・過剰在庫・輸送遅延を早期検知する。

### 2.2 技術スタック

| 層 | 技術 |
|----|------|
| バックエンド | Python 3.x / FastAPI / SQLAlchemy / PostgreSQL |
| フロントエンド | Next.js 14+ / TypeScript / Tailwind CSS |
| 認証 | JWT（python-jose）+ bcrypt |
| スケジューリング | APScheduler（5分間隔バッチ） |
| マイグレーション | Alembic（11バージョン管理済） |
| インフラ | Docker Compose（開発環境） |

### 2.3 データモデル全体図

```

           products
```

## 3. 利用者ロールと権限設計

### 3.1 ロール定義

#### ロール1：実務者（Operator）

日々の物流業務を遂行するオペレーション担当者。工場・DC・TC・配送部門に所属する現場担当者を想定する。

**担当範囲の制御**:
- `assigned_location_ids`：担当拠点IDのカンマ区切りリスト（DBに保持）
- `assigned_category_ids`：担当商品カテゴリのカンマ区切りリスト（DBに保持、**現行は未活用**）
- 担当拠点外のデータは参照不可（APIレベルで制御）

#### ロール2：管理者（Administrator）

システム運用上の管理責任者。全拠点・全商品・全データへのアクセス権限を持つ。

### 3.2 権限マトリクス（現行実装）

## 4. 業務フロー定義

### 4.1 在庫管理フロー

```
[現状確認]
実務者 → 在庫照会画面（担当拠点）
        → 在庫量・安全在庫・賞味期限を確認
        → アラートバッジで異常を把握

[棚卸修正]
実務者 → 在庫修正フォーム（数量・賞味期限・備考）
        → 確認ダイアログ → 確定
        → 監査ログ自動記録
        → アラート自動再評価

[一括更新]
実務者 → CSVアップロード
        → バリデーション → プレビュー確認 → 適用
        → 監査ログ自動記録
```

**現行実装の課題**：在庫の「現在値スナップショット」のみ管理。入庫・出庫の個別明細履歴がない。

### 4.2 発注・補充フロー

```
[発注作成]
実務者 → 発注入力（補充先・補充元・商品・数量・希望納期）
        → プレビュー確認（リードタイム・コスト見積含む）
        → 発注確定（status: DRAFT → CONFIRMED）
        → 監査ログ記録

[発注追跡]
実務者 → 発注一覧（ステータス別フィルタ）
        → ステータス手動更新（IN_TRANSIT / DELIVERED / CANCELLED）

[入荷確認]
実務者 → 発注ステータスを DELIVERED に手動更新
```

**現行実装の課題**：
- 発注確定後の配送記録が自動生成されない（手動で別途作成が必要）
- 入荷確認（DELIVERED）しても在庫が自動加算されない
- 発注ステータス遷移の整合性チェックが不十分

### 4.3 配送管理フロー

```
[配送記録作成]
実務者 → 配送記録入力（ルート・出発/到着予定日・数量）
        → 配送記録確定（status: SCHEDULED）

[輸送追跡]
実務者 → 配送一覧（ステータス・遅延フラグ確認）
        → ステータス手動更新（DEPARTED / IN_TRANSIT / ARRIVED）
        → 遅延時：遅延理由入力

[到着確認]
実務者 → ステータスを ARRIVED に更新
```

**現行実装の課題**：
- 到着確認（ARRIVED）しても在庫が自動加算されない
- 発注と配送記録の紐付け（order_id）は手動であり、連動が不完全

### 4.4 アラート対応フロー

```
[アラート検知]
APScheduler（5分間隔） → 全在庫評価
        → 在庫切れ・安全在庫割れ・過剰在庫・賞味期限アラート自動生成
        → 既存アラートの自動解消

[アラート確認]
実務者 → 通知ベル（バッジ件数）→ アラート一覧
        → severity（danger/warning）・拠点・商品を確認

[アラート対応]
実務者 → 対応ステータス更新（OPEN → IN_PROGRESS → RESOLVED）
        → 発注・在庫修正などの対応作業を並行実施
```

**現行実装の課題**：
- アラート対応の際、実際に取った対処（どの発注を起こしたか等）との紐付けがない
- 遅延アラートは配送記録のDELAYEDステータスとアラートが連動していない

### 4.5 マスタデータ管理フロー

```
[マスタ登録・更新]
管理者 → マスタ一覧（拠点/商品/ルート/ユーザー）
        → 新規登録または編集
        → 確認ダイアログ → 確定
        → 監査ログ自動記録

[マスタ廃止]
管理者 → 論理削除（is_active = false）
        → 過去トランザクションデータは参照可能なまま保持

[安全在庫設定]
管理者 → 安全在庫設定画面（拠点×商品マトリクス）
        → safety_stock / max_stock を直接編集
        → 保存 → アラート閾値が即時反映
```

## 5. 機能要件（現行実装）

### 5.1 実務者向け機能

#### OP-01：ダッシュボード（実務者）

- 担当拠点のアラートサマリー（severity別件数）を表示
- 在庫アラート（安全在庫割れ商品数）を表示
- **▲ 部分実装**：工場→DC→TC→小売のフロー図（輸送中数量・到着予定）が未表示

#### OP-02：アラート管理

- アラート一覧（severity/status/拠点でフィルタ）
- 対応ステータス管理（OPEN / IN_PROGRESS / RESOLVED）
- アラートタイプ：STOCKOUT / LOW_STOCK / OVERSTOCK / EXPIRY_EXPIRED / EXPIRY_NEAR / DELAY / CUSTOM
- 自動生成アラートと手動アラートの区別
- **✅ 実装済**

#### OP-03：在庫照会・修正

- 在庫一覧（担当拠点のみ、商品名・拠点名・カテゴリ検索）
- 在庫数量・賞味期限・備考の手動修正（確認ダイアログあり）
- 安全在庫・最大在庫に対する充足度の視覚的表示
- **✅ 実装済**

#### OP-04：発注・補充指示

- 発注タイプ：REPLENISHMENT（補充）/ TRANSFER（移管）/ EMERGENCY（緊急）
- 発注入力フォーム（補充元・補充先・商品・数量・希望納期）
- 2ステップ確認：入力 → プレビュー（リードタイム・コスト見積）→ 確定
- 発注コード自動採番（ORD-YYYYMMDD-NNN形式）
- ステータス管理：DRAFT / CONFIRMED / IN_TRANSIT / DELIVERED / CANCELLED
- **✅ 実装済**

#### OP-05：配送計画確認・調整

- 配送記録一覧（拠点・ステータス・日付範囲でフィルタ）
- 配送記録作成（ルート・出発予定・到着予定・数量）
- ステータス更新：SCHEDULED / DEPARTED / IN_TRANSIT / ARRIVED / DELAYED / CANCELLED
- 遅延理由記録フィールド
- 配送コード自動採番（DLV-YYYYMMDD-NNN形式）
- **✅ 実装済**

#### OP-06：中期計画ビュー

- 発注・配送データを集計した週別計画グリッド表示
- **▲ 部分実装**：UIページは存在するが表示データが限定的

#### OP-07：データアップロード

- CSV形式による在庫データ一括更新
- アップロード前バリデーション（フォーマット・数値範囲・拠点/商品の存在確認）
- プレビュー表示（変更予定データの確認）→ 適用
- バリデーションエラーはエラー行・列・理由を表示
- **✅ 実装済**

#### OP-08：操作履歴確認

- 実務者が自身の操作ログを参照（監査ログAPIの自己フィルタ）
- 日付範囲・操作種別でフィルタ
- **✅ 実装済**

## 6. 非機能要件

### 6.1 パフォーマンス

| 指標 | 目標値 | 現行実装 |
|------|--------|----------|
| ダッシュボード初期表示 | 3秒以内 | SQLAlchemy joinedload で N+1 回避済 |
| 在庫データ更新反映 | 最大5分以内 | APScheduler 5分間隔で対応 |
| CSVアップロード（1万行以内） | 30秒以内 | pandas 同期処理（大量データは要対応） |
| API応答時間（通常） | 1秒以内 | DBコネクションプール（pool_size=10）で対応 |

### 6.2 可用性

- 稼働率目標：99.0%以上（月次）
- 計画メンテナンス時間を除く
- 現行：単一インスタンス構成（スケールアウト未対応）

### 6.3 セキュリティ

| 項目 | 要件 | 現行実装 |
|------|------|----------|
| 通信暗号化 | HTTPS必須 | 開発環境はHTTP（本番でNginx + SSL必要） |
| パスワード保存 | bcryptハッシュ | ✅ 実装済 |
| セッション管理 | JWT 30分タイムアウト | ✅ 実装済 |
| 認証情報保存場所 | httpOnly Cookie 推奨 | ▲ 現行はlocalStorage（XSSリスクあり） |
| ログイン試行制限 | ブルートフォース対策 | ❌ 未実装 |
| 監査ログ保持期間 | 最低1年 | ⚠️ 保持期間ポリシー未定義 |
| APIドキュメント | 本番では非公開 | ⚠️ /docs, /redoc が有効のまま |

### 6.4 保守性

- バックエンド：3層構成（API層 → サービス層 → モデル層）
- フロントエンド：ページ / コンポーネント / ユーティリティの責務分離
- マイグレーション：Alembic による履歴管理（11バージョン）
- 論理削除の徹底（物理削除なし）

| ユーザーマスタ | users | username, email, role, assigned_location_ids, assigned_category_ids | 管理者のみ |
| KPI閾値マスタ | kpi_thresholds | kpi_key, label, warning_value, danger_value, unit | 管理者のみ |

### 7.2 トランザクションデータ（現行実装）

| データ種別 | テーブル | 主要項目 | 更新者 |

## 8. 権限・セキュリティ設計

### 8.1 API認証・認可フロー

```
クライアント → Bearer Token（JWT） → FastAPI Depends
    ↓
get_current_user() → JWTデコード → users テーブル照合
    ↓
require_administrator() or check_location_access()
    ↓
OK: 処理実行 / NG: HTTP 403 Forbidden
```

### 8.2 マスタ変更の安全設計

- **変更確認の2ステップ**：一覧→詳細→編集→確認ダイアログ→確定
- **変更履歴の自動記録**：全操作を audit_logs に記録（変更者・日時・詳細）
- **論理削除の徹底**：`is_active = false` で廃止。物理削除は行わない
- **廃止データの参照保持**：廃止済マスタを参照するトランザクションデータは引き続き参照可能

### 8.3 データアップロードの安全設計

- ファイル形式・サイズ制限：CSV/Excel、最大10MB
- ヘッダー行一致確認・必須項目の存在確認
- 数値項目の範囲チェック（負数・異常値の検出）
- 拠点ID・商品IDの存在確認（マスタ照合）
- 原子性の保証：エラーがある場合は全件中断（部分取り込み禁止）
- バリデーションエラーはエラー行・列・理由を一覧表示

## 9. 業務フロー観点からの改修ポイント

本章は、第4章「業務フロー定義」に基づくレビュー結果として、**業務上の連続性・整合性が欠けている箇所**を改修ポイントとして整理したものである。

---

### 【改修ポイント 1】配送到着時の在庫自動反映

**分類**: 業務フロー断絶（重大度：高）

**現状の問題**:
配送記録のステータスを `ARRIVED` に更新しても、配送先（to_location）の在庫数量が自動加算されない。実務者は配送到着後に在庫修正画面で手動更新する必要があり、更新漏れが発生するリスクがある。

**理想の業務フロー**:
```
配送ステータス ARRIVED に更新
    → 自動的に delivery_records.quantity 分を
       to_location の inventories.quantity に加算
    → アラート自動再評価
    → 監査ログ記録（action='DELIVERY_ARRIVED', resource='inventory'）
```

**改修内容**:
- バックエンド：`PATCH /api/deliveries/{id}/status` で `status=ARRIVED` の場合、inventories を自動更新する処理を追加
- フロントエンド：到着確認ボタンに「在庫を自動更新します」の説明文を追加

---

### 【改修ポイント 2】発注確定時の配送記録自動生成

**分類**: 業務フロー断絶（重大度：高）

**現状の問題**:
発注を確定（CONFIRMED）しても、対応する配送記録（delivery_records）は自動生成されない。実務者は発注と配送の2画面で別々に入力する必要があり、二重入力の手間が生じると同時に入力漏れのリスクがある。

**理想の業務フロー**:
```
発注確定（status: CONFIRMED）
    → ルートマスタから対応ルートを自動検索
       （from_location_id + to_location_id で照合）
    → 配送記録を自動生成（status: SCHEDULED）
       expected_arrival_date = requested_date + lead_time_days
    → 実務者は配送記録を確認・調整するだけ
```

**改修内容**:
- バックエンド：発注確定APIに配送記録自動生成ロジックを追加
- ルートが存在しない場合はエラー or 手動作成を促すメッセージを返す
- フロントエンド：発注確定後に「配送記録を作成しました」の通知を表示

---

### 【改修ポイント 3】在庫移管（TRANSFER）時の出荷元在庫自動減算

**分類**: 業務フロー断絶（重大度：高）

**現状の問題**:
発注タイプが `TRANSFER`（拠点間移管）の場合でも、出荷元（from_location）の在庫が自動減算されない。移管発注では「元拠点の在庫を減らして、先拠点に届ける」という物理的な動きがあるが、システム上では在庫移動が記録されない。

**改修内容**:
- バックエンド：TRANSFER型発注が CONFIRMED になった時点で from_location の inventories.quantity を即時減算
- または配送記録の DEPARTED ステータス更新時に減算（運用ポリシーに応じて選択）
- 在庫が不足する場合は発注プレビュー時にエラーを返す

---

### 【改修ポイント 4】入出庫明細トランザクションの欠如

**分類**: データ完全性・監査対応（重大度：高）

**現状の問題**:
`inventories` テーブルは現在庫数量のスナップショットのみを保持している。在庫がいつ・何の理由で・どれだけ増減したかの明細（入出庫トランザクション）が記録されない。

**業務上の影響**:
- 棚卸差異の原因追跡ができない
- 発注・配送・手動修正のどれで在庫が変動したかが不明
- 会計・コンプライアンス観点での在庫変動証跡が不足

**改修内容**:
- 新テーブル `inventory_transactions` の追加
  ```
  id, inventory_id, transaction_type (ORDER_IN/DELIVERY_ARRIVED/MANUAL_ADJUST/CSV_UPLOAD/TRANSFER_OUT/TRANSFER_IN),
  quantity_change, quantity_after, reference_id (order_id or delivery_id), note,
  created_by, created_at
  ```
- 在庫が変動するすべての処理（手動修正・CSV適用・配送到着・発注移管）でトランザクションを記録
- 管理者向けに在庫変動履歴ビューを追加

---

### 【改修ポイント 5】KPIダッシュボードの実績値計算

**分類**: 機能不足（重大度：中）

**現状の問題**:
管理者ダッシュボードは拠点数・商品数・ルート数・ユーザー数のマスタ統計のみを表示している。KPI閾値設定機能（AD-07）は実装済みだが、対応するKPI実績値の計算・表示が実装されていない。

**未実装のKPI指標**:

| KPI名 | 計算式 | データソース |
|-------|--------|-------------|
| 欠品率 | 在庫ゼロ商品数 / 全商品数 × 100 | inventories |
| 安全在庫割れ率 | quantity < safety_stock の件数 / 全件数 × 100 | inventories |
| 在庫回転率 | 期間出荷量 / 期間平均在庫量 | inventories + delivery_records |
| 配送遅延率 | DELAYED件数 / 全配送件数 × 100 | delivery_records |
| 発注充足率 | DELIVERED件数 / CONFIRMED件数 × 100 | orders |
| 過剰在庫率 | quantity > max_stock の件数 / 全件数 × 100 | inventories |

**改修内容**:
- バックエンド：KPI集計エンドポイント `GET /api/dashboard/kpi` を新規追加
- kpi_thresholds の warning/danger 値と実績値を比較してステータスを返す
- フロントエンド：管理者ダッシュボードにKPIカード群（数値 + 色分けバッジ）を追加

---

### 【改修ポイント 6】商品カテゴリ別アクセス制御の未活用

**分類**: 権限設計の不完全性（重大度：中）

**現状の問題**:
ユーザーテーブルに `assigned_category_ids` フィールドが存在するが、APIの権限チェックで一切使用されていない。拠点は制御されているが、商品カテゴリによるアクセス制限が機能していない。

**業務上の影響**:
「A社担当で飲料カテゴリのみ」「B社担当でアルコールカテゴリのみ」といった商品ベースの権限分離が実現できていない。

**改修内容**:
- バックエンド：在庫・発注・配送の一覧取得APIに `assigned_category_ids` フィルタを追加
- ユーザー管理画面：カテゴリ割当をマスタ商品カテゴリから選択できるUI改善

---

### 【改修ポイント 7】パスワード変更機能の欠如

**分類**: セキュリティ（重大度：中）

**現状の問題**:
ユーザーがログイン後に自分のパスワードを変更する機能がない。初期パスワードの変更や定期的なパスワード更新ができず、セキュリティリスクが高い。

**改修内容**:
- バックエンド：`POST /api/auth/change-password`（現在のパスワード + 新しいパスワードで認証後に更新）
- フロントエンド：実務者・管理者共通のプロフィール/パスワード変更ページ

---

### 【改修ポイント 8】ログイン試行制限の欠如

**分類**: セキュリティ（重大度：中）

**現状の問題**:
ログイン試行に対するレート制限が実装されていない。ブルートフォース攻撃によるパスワード総当たりが可能な状態。

**改修内容**:
- バックエンド：`slowapi` または `fastapi-limiter` によるレート制限を追加
  - 推奨：同一IPから5回失敗で30分ロックアウト
- 失敗ログを audit_logs に記録（action='LOGIN_FAILED'）

---

### 【改修ポイント 9】業務カレンダー（営業日管理）の未実装

**分類**: 機能不足（重大度：中）

**現状の問題**:
発注のプレビュー時にリードタイム（lead_time_days）を加算して納期を計算しているが、土日・祝日・夏季休業等の非営業日が考慮されていない。例：リードタイム2日の発注を金曜日に行った場合、実際の到着は翌月曜日以降だが、日曜日と計算される。

**初版要件定義（§7.1）に「業務カレンダー」マスタが定義されているが未実装のまま。**

**改修内容**:
- 新テーブル `business_calendars`（date, is_business_day, description）
- ルートのリードタイム計算ユーティリティ：`calc_delivery_date(start_date, lead_time_days)` → 営業日のみカウント
- 管理者向け業務カレンダー設定画面

---

### 【改修ポイント 10】CSVアップロード履歴の未保持

**分類**: 運用・監査（重大度：低）

**現状の問題**:
CSVアップロードの実行記録が監査ログ（audit_logs）に一行記録されるが、アップロード内容の詳細・バリデーション結果・エラー内容が保持されない。後から「いつ・誰が・どんなデータをアップロードし、何件成功・何件失敗したか」を確認できない。

**改修内容**:
- 新テーブル `upload_histories`（user_id, filename, total_rows, success_rows, error_rows, error_detail_json, status, created_at）
- アップロード処理実行時に履歴を記録
- 管理者・実務者向けにアップロード履歴参照ページを追加

---

### 【改修ポイント 11】在庫推移グラフの欠如

**分類**: 可視化・意思決定支援（重大度：低）

**現状の問題**:
在庫の「現在値」のみ表示され、過去の推移グラフがない。在庫が増えているのか減っているのかのトレンドが視覚的に把握できない。

**前提条件**：改修ポイント4（入出庫明細トランザクション）の実装が必要。

**改修内容**:
- 在庫照会画面に選択商品の在庫推移チャートを追加（直近30日等）
- バックエンド：`GET /api/inventory/{id}/history` エンドポイント追加

---

### 【改修ポイント 12】認証トークン保存場所の脆弱性

**分類**: セキュリティ（重大度：中）

**現状の問題**:
JWTトークンを `localStorage` に保存しているため、XSS（Cross-Site Scripting）攻撃によりトークンが窃取されるリスクがある。

**改修内容**:
- `httpOnly Cookie` + `Secure` 属性に移行（JavaScriptからアクセス不可）
- CSRF対策として CSRF トークンを実装
- フロントエンドの `apiFetch` をCookie認証に対応させる

---

### 【改修ポイント 13】本番環境向けセキュリティ設定

**分類**: セキュリティ（重大度：高）

**現状の問題**:
- APIドキュメント（`/docs`, `/redoc`）が本番環境でも公開状態になっている
- `SECRET_KEY` がデフォルト値のまま運用されるリスクがある
- CORS設定がワイルドカード（`*`）のままになっている可能性がある

**改修内容**:
- `ENVIRONMENT=production` の場合、`docs_url=None, redoc_url=None` を設定
- `SECRET_KEY` の本番環境設定ガイドを整備（`openssl rand -hex 32` で生成）
- CORS `ALLOWED_ORIGINS` を本番URLのみに限定

---

### 【改修ポイント 14】実務者向けダッシュボードのフロー可視化

**分類**: UI/UX（重大度：低）

**現状の問題**:
初版要件定義（OP-01）で要求されていた「工場→DC→TC→小売のフロー図（各段階の輸送中数量・到着予定）」が未実装。現在のダッシュボードはアラート件数と在庫サマリーのみ表示。

**改修内容**:
- フロー状況サマリー：`IN_TRANSIT` または `DEPARTED` 状態の配送記録を拠点別に集計
- ダッシュボードにサプライチェーン段階別の輸送中件数・到着予定を表示

---

### 【改修ポイント 15】中期計画ビューの実装完成

**分類**: 機能不足（重大度：低）

**現状の問題**:
`/operator/planning`（中期計画ビュー）のページは存在するが、発注・配送データを集計した週別グリッド表示のロジックが完全に実装されていない。

**改修内容**:
- バックエンド：`GET /api/planning/weekly?weeks=4` エンドポイント追加
  - 週単位で予定発注量・予定入荷量・推定在庫を集計
  - シナリオの demand_factor を適用したシミュレーション値も返す
- フロントエンド：週×商品グリッド、安全在庫割れ予測セルのハイライト
