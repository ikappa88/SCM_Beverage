## 1. 本文書の目的と方針

### 1.1 目的

本文書は、v2レビューで挙がった「業務フロー観点の改修ポイント」を引き継ぎつつ、**実務担当者（オペレーター）が実際に毎日システムを使う際に感じる不満・迷い・ミスのリスク**を洗い出し、次期開発フェーズの要件として定義するものである。

技術的な正しさではなく、**「人間がシステムと対話する場面」**に着目する。

### 1.2 レビュー対象ロール

本書では主に以下を対象とする。

| ロール | 想定人物像 | 利用端末 |
|--------|-----------|---------|
| 実務担当者（オペレーター） | DC・TC・小売在庫担当。毎日在庫確認・発注判断を行う | PC（主）／倉庫内タブレット（副） |
| 管理者（アドミニ） | 複数拠点を統括するロジスティクスマネージャー | PC |

### 1.3 評価方法

以下のシナリオで画面を実際に辿り、「困ること」を都度記録した。

- 朝の始業時：アラート確認 → 対応判断 → 発注作成
- 日中：配送追跡 → 到着確認 → 在庫更新
- 月次：CSVアップロードによる棚卸し反映

## 2. 実務者レビュー：「1日の業務」シナリオウォークスルー

### シナリオA：朝のアラート対応（推定所要時間 30分）

```
現行フロー：
[ダッシュボード確認] → アラート5件の赤バッジを発見
→ [アラートページ] に移動
→ アラート詳細を確認（"東京DC / コーラ500ml / 在庫不足"）
→ 「どのくらい不足しているか確認したい」
→ [在庫ページ] に移動 ← ここでアラート画面を離れる
→ 東京DCで検索 → コーラ500mlを探す
→ 残40ケース・安全在庫100ケースと確認
→ 「発注しよう」
→ [発注ページ] に移動 ← 在庫画面も閉じる
→ 発注入力（商品・数量・ルートを手で入力）
→ 「どのルートが使えるか分からない」
→ [ルートページ（参照）] を別タブで開く ← タブが増える
→ ルートを確認して発注ページに戻る → 入力再開
→ 発注完了
→ 「アラートを対応中にしなきゃ」
→ [アラートページ] に戻る → 対象アラートを探す
→ ステータスを「対応中」に更新
→ 「あれ、さっき何件アラートがあって何件対応したっけ？」
```

**問題点の数**: このシナリオだけで **7回のページ遷移** が発生し、ユーザーは手作業で情報をつなぎ合わせている。

---

### シナリオB：配送到着後の在庫反映（推定所要時間 15分）

```
現行フロー：
[配送追跡ページ] → 「大阪DC行き / コーラ / 200ケース」の配送が到着
→ ステータスを「到着済み」に更新
→ （ここでシステム上の作業は完了と思いがち）
→ 実は在庫ページで手動更新が必要（システムは自動連携しない）
→ 翌日、在庫アラートが「在庫不足」で再発火
→ 「あれ？昨日入荷したのに」→ 在庫更新忘れに気づく
→ 手動で在庫を +200ケース 修正
```

**問題点**: 配送到着と在庫反映が連携していないため、**作業漏れ・二重入力が構造的に発生する**。
実務者は「配送到着 = 在庫が増える」と自然に認識するが、システムはそう動かない。

---

### シナリオC：棚卸しCSVアップロード（推定所要時間 45分）

```
現行フロー：
[アップロードページ] → テンプレートをダウンロード
→ Excelで棚卸しデータ入力（拠点コード・商品コードを手打ち）
→ 「商品コードが何だったか分からない」
→ [商品マスタページ] を開いて確認 ← 別タブ必須
→ Excelに戻って入力続行
→ アップロード → プレビュー確認 → コミット
→ 「どこが変更されたか画面に残らない」
→ アップロード後に在庫ページへ移動して目視確認
```

**問題点**: アップロードに必要な情報（コード類）が別画面にあるため、実務者は常に複数タブを維持する必要がある。

## 3. 画面横断UX課題（詳細）

### 3.1 アラート→対応アクション フロー断絶

**現状の問題**

アラートを確認してから実際の対応（発注・在庫修正）まで、画面を3〜4回往復する必要がある。その間、「どのアラートへの対応か」という文脈が失われる。

| 問題 | 影響 |
|------|------|
| アラートから在庫詳細へのリンクがない | 別ページで商品・拠点を再検索する手間 |
| アラートから発注作成へのショートカットがない | 発注ページで商品・ルートを手入力 |
| 発注後にアラートステータスが自動更新されない | 対応完了を手動で2箇所に記録 |
| アラートと対応アクションが紐付かない | 「このアラートに対してどんな対応をしたか」が追えない |

**具体的に困ること**

- アラート「東京DC / コーラ500ml / 在庫不足」を見て発注したが、6時間後に同じアラートが再発。発注済みであることをシステムが知らないため。
- 管理者から「このアラート、なんで未対応なの？」と言われたが、実際には発注済みで追跡できない。

---

### 3.2 発注→配送→在庫 自動連携の欠落

**現状の問題**（v2でも指摘済み・未改修）

| 業務フロー | 現状 | あるべき姿 |
|-----------|------|-----------|
| 発注確認 → 配送記録作成 | 手動で別途作成が必要 | 発注確認時に自動生成 |
| 配送到着 → 在庫加算 | 手動で在庫ページを更新 | 到着ステータス変更で自動加算 |
| 転送発注 → 出荷元在庫減 | 自動減算されない | 発注確認時に自動減算 |

**実務者が受ける影響**

- 発注と配送と在庫を「3つの別帳簿」として管理する羽目になる
- どれか1つ更新漏れすると、データ不整合が起きてもシステムが検知しない
- 夕方の業務終了時に「今日の入力が完了しているか」を一覧で確認する手段がない

---

### 3.3 情報の分散と「記憶頼り」問題

**発注作成時に必要な情報がバラバラ**

発注を1件作成するためには以下を知っている必要がある：

| 必要情報 | どこで確認できるか |
|---------|-----------------|
| 現在在庫数 | 在庫ページ（別画面） |
| 安全在庫・最大在庫 | 在庫ページ（別画面）または管理者の安全在庫設定画面 |
| 使用可能ルート | ルート参照ページ（別画面） |
| リードタイム | ルート参照ページ（別画面）← 発注プレビューで見られるが入力後 |
| 最小発注数量 | 商品マスタページ（別画面） |
| 直近の発注履歴 | 発注ページ一覧（同ページだが一覧に埋もれる） |

**実務者の行動パターン**

> 「毎朝同じルートで同じ商品を発注するから、数値を頭で覚えてる。新人が入ったとき、どこを見ればいいか教えるのが大変だった。」

---

### 3.4 コンテキスト消失（ページ遷移で迷子になる）

**現状のナビゲーション設計の問題**

- ページ間に「戻る」ための文脈リンクがない
  - アラートページ → 在庫ページ → 発注ページと進んだとき、元のアラートへ戻れない
- URLパラメータやフィルタ状態がページ遷移で失われる
  - 在庫ページで「東京DC」でフィルタしていても、別ページ経由で戻ると初期状態に戻る
- ダッシュボードの「危険アラート 5件」をクリックしても、アラートページにフィルタが引き継がれない

**実務者が感じる不便**

> 「在庫一覧を拠点でフィルタして確認中に、発注のために別ページに行ったら、戻ってきたときにフィルタがリセットされてた。毎回やり直し。」

---

### 3.5 アラート疲弊と優先順位不明

**現状の問題**

| 問題 | 詳細 |
|------|------|
| 同一アラートの繰り返し発火 | 5分ごとに評価され、対応中でも同じアラートが再生成される可能性 |
| スヌーズ・一時停止機能がない | 「明日入荷予定なので今は無視したい」ができない |
| 優先度の並び替えが甘い | 欠品（STOCKOUT）と期限切れ（EXPIRY_EXPIRED）が混在し、何を先に対応すべきか不明確 |
| アラートと在庫数・売上の関係が見えない | 「欠品だが売れない商品」と「欠品かつ需要大」の区別ができない |
| 通知ベルにアラート件数しか出ない | 緊急度・拠点情報が分からず、クリックするまで内容不明 |

**実務者が感じる不便**

> 「朝来たら通知が30件ある。でも何が本当に急ぎかわからないから全部開いてみないといけない。10件は昨日から変わってないアラートの再通知だった。」

---

### 3.6 モバイル・現場作業への非対応

**現状の問題**

- レスポンシブデザインが未整備（PCレイアウト前提）
- 倉庫内でタブレットを使った棚卸し中に在庫数を更新する動作が困難
- 配送担当者がスマートフォンで配送ステータスを更新できない
- CSV作成のためにExcelが必要（倉庫PCがない場合の代替手段がない）

**実務者が感じる不便**

> 「倉庫の棚で数えながらシステムに入力したいのに、PCのある事務室に戻ってからじゃないと更新できない。その間に数字を紙に書いて転記している。二度手間。」

---

### 3.7 管理者・実務者間の情報非対称

**現状の問題**

| 問題 | 影響 |
|------|------|
| 安全在庫は管理者画面でしか設定・確認できない | 実務者は「何ケースで警告が出るか」を知らずに作業 |
| 発注ルールや発注基準が画面上に存在しない | 口頭・メモで補完、引継ぎ時に漏れる |
| 管理者が閲覧できる発注・配送は Read-Only | 管理者が問題を発見しても、実務者に通知する手段がない |
| シナリオ・KPIしきい値の変更が実務者に見えない | 突然アラートの頻度が変わっても理由が分からない |

---

### 3.8 補充元在庫の不可視問題（権限設計の根本的矛盾）

**現状の問題**

現行の権限設計では「自拠点のみ在庫閲覧・編集可」になっている。しかし発注・補充の業務フローでは、**注文を送る相手（補充元）の在庫が見えなければ発注量を判断できない**。

```
サプライチェーン構造と発注方向：

工場（Factory）
  ↑ DC が補充発注
広域物流センター（DC）
  ↑ TC が補充発注
地域配送センター（TC）
  ↑ 小売が補充発注
小売チャネル（Retail）
```

神戸TCのオペレーターが「大阪DCに補充発注しよう」と思っても、**大阪DCの在庫が見えないため以下の問題が生じる**：

| 問題 | 影響 |
|------|------|
| 補充元が欠品・低在庫でも気づかず発注 | フルフィルできない発注が確定してしまう |
| 発注数量を根拠なく決定 | 過少発注（何度も発注）または過剰発注が発生 |
| 「なぜ届かないのか」が分からない | 補充元の在庫不足を発注後に初めて知る |

**設計上の対立**

| 観点 | 現行設計の意図 | 実務上の要求 |
|------|--------------|-------------|
| セキュリティ | 他拠点情報を隔離する | 補充元の在庫は発注判断に必須 |
| 権限管理 | 自拠点のみ編集・閲覧 | 上流拠点は**読み取り専用**で十分 |
| 業務フロー | - | 「注文できる相手の在庫量」を知る権利は自明 |

**あるべき設計**

> 「ルートが存在する = そこから注文できる = そこの在庫を確認できる」
> 閲覧権限と編集権限を分離することで、セキュリティを保ちながら業務上の可視性を確保する。

## 4. 新機能要件定義

### P1：必須要件（業務が止まる / 重大なミスが起きる）

---

#### P1-001：配送到着→在庫自動反映

**要件ID**: P1-001
**対応課題**: 3.2（v2改修ポイント①）
**概要**: 配送ステータスを「到着済み（ARRIVED）」に変更した時点で、宛先拠点の在庫数量を自動的に加算する。

**業務シナリオ**:
1. 実務者が配送追跡ページで配送を「到着済み」に更新
2. システムが宛先拠点 × 該当商品の在庫に数量を加算
3. 在庫変更履歴に「配送到着による自動加算（配送コード）」として記録
4. 関連するアラート（低在庫/欠品）を再評価し、解消されていれば自動でRESOLVEDにする

**受け入れ条件**:
- 配送 ARRIVED 時に在庫が正確に加算されること
- 手動の在庫更新は不要になること（任意で手動修正は可能）
- 自動更新された旨が在庫履歴に残ること

---

#### P1-002：発注確定→配送記録自動生成

**要件ID**: P1-002
**対応課題**: 3.2（v2改修ポイント②）
**概要**: 発注を「確定（CONFIRMED）」にした時点で、対応する配送記録を自動生成する。

**業務シナリオ**:
1. 実務者が発注プレビューを確認し「確定」を押す
2. システムが以下の配送記録を自動生成:
   - 出発地 = 発注の from_location
   - 到着地 = 発注の to_location
   - 商品・数量 = 発注の商品・数量
   - ステータス = SCHEDULED
   - 予定出発日 = 発注日 + ルートのリードタイム
   - 関連発注ID = 当該発注のID
3. 配送記録に「発注（ORD-XXXX）から自動生成」と表示される

**受け入れ条件**:
- 発注確定後に配送ページへ移動すると自動生成された配送記録が存在する
- 配送記録から元の発注レコードにリンクで遷移できる
- 発注ページから対応する配送記録にリンクで遷移できる

---

#### P1-003：転送発注時の出荷元在庫自動減算

**要件ID**: P1-003
**対応課題**: 3.2（v2改修ポイント③）
**概要**: 転送タイプ（TRANSFER）の発注が確定した時点で、出荷元拠点の在庫数量を減算する。

**業務シナリオ**:
1. 実務者が「転送」タイプで発注確定
2. システムが出荷元拠点 × 商品の在庫を発注数量分減算
3. 在庫変更履歴に「転送発注による出荷（発注コード）」として記録
4. 在庫変更後に安全在庫アラートを再評価する

**受け入れ条件**:
- 転送発注確定時に出荷元在庫が減算されること
- 在庫不足（発注数量 > 現在在庫）の場合は警告を表示し、確認後に続行できる
- 補充（REPLENISHMENT）・緊急（EMERGENCY）タイプでは減算しない

---

#### P1-004：アラートから直接アクションへのクイックリンク

**要件ID**: P1-004
**対応課題**: 3.1
**概要**: アラート一覧・詳細から、関連する在庫・発注作成・配送詳細へ1クリックで遷移できるようにする。

**業務シナリオ**:
1. 実務者がアラート「東京DC / コーラ500ml / 在庫不足」を確認
2. アラートカードに以下のボタン/リンクを表示:
   - [在庫を確認する] → 在庫ページを当該拠点・商品でフィルタした状態で開く
   - [発注を作成する] → 発注作成ページを当該商品・推奨数量で事前入力した状態で開く
3. 発注作成後、「アラートを対応中にする」プロンプトを表示（任意で実行可）

**UI仕様**:
- アラートカードに「在庫確認」「発注作成」ボタンを追加（低在庫・欠品アラートのみ）
- 期限切れアラートは「在庫確認」のみ（期限日修正へリンク）
- 遅延アラートは「配送詳細を確認」ボタン

---

#### P1-005：アラートと対応アクションの紐付け記録

**要件ID**: P1-005
**対応課題**: 3.1
**概要**: アラートに対してどの発注・在庫修正が行われたかを記録・表示する。

**業務シナリオ**:
1. アラートから発注作成ページに遷移
2. 発注確定時、「このアラート（ALT-XXX）への対応として発注を作成」フラグを設定可能
3. アラート詳細画面に「対応履歴」セクションを追加:
   - 紐付けられた発注番号
   - 在庫変更履歴
   - ステータス変更履歴

**受け入れ条件**:
- アラート詳細画面で対応アクションの一覧が確認できること
- 対応アクションにリンクで遷移できること

---

---

#### P1-006：ルートベースの補充元在庫可視化（読み取り専用）

**要件ID**: P1-006
**対応課題**: 3.8
**概要**: ルートマスタで「FROM → 自拠点」として定義されている拠点（補充元）の在庫を、読み取り専用で閲覧できるようにする。編集権限は与えない。

**権限ルール（変更後）**:

| 拠点との関係 | 閲覧 | 編集 |
|------------|------|------|
| assigned_location_ids（自拠点） | ✅ | ✅ |
| ルートの供給元（FROM拠点） | ✅（読み取り専用） | ❌ |
| それ以外の拠点 | ❌ | ❌ |

**業務シナリオ**:
1. 神戸TCのオペレーターが発注作成画面を開く
2. 「補充元：大阪DC」「商品：コーラ500ml」を選択
3. 画面上に大阪DCの現在在庫が読み取り専用で表示される：
   ```
   [補充元：大阪DC の在庫状況]
    コーラ 500ml: 320ケース（安全在庫: 200ケース ✅ 余裕あり）
   ```
4. 在庫を確認したうえで発注数量を決定する

**UI仕様**:

- 補充元の在庫が安全在庫を下回っている場合は警告表示（「この拠点の在庫が少ないため補充が遅れる可能性があります」）

**受け入れ条件**:

**バックエンド実装方針**:
```

  現行: assigned_location_ids のみ返却

  ・assigned_location_ids → 通常返却（編集可）

upstream_location_ids の算出:
  SELECT DISTINCT r.origin_id
  FROM routes r
  WHERE r.destination_id IN (user.assigned_location_ids)
    AND r.is_active = true
```

#### P1-007：利用可能在庫（ATP: Available to Promise）の表示

**要件ID**: P1-007
**対応課題**: 3.8
**概要**: 補充元の「現在庫数」だけでなく、**他拠点からの確定発注で既に引き当て済みの数量を除いた「実際に注文可能な数量」**を表示する。

**問題の背景**:

> 大阪DCの現在庫が320ケースでも、すでに他のTCから合計300ケースの確定発注が入っていれば、実際に補充できるのは20ケースしかない。これを知らずに200ケース発注すると確実に不足する。

**ATP計算式**:
```
ATP（利用可能在庫）
  = 現在庫数
  - 確定済み発注の未出荷数量（status: confirmed または in_transit）
  + 入荷予定数量（自拠点への配送で status: scheduled または in_transit のもの）
```

**UI仕様**:
```
[補充元：大阪DC の在庫状況]
 コーラ 500ml
  現在庫:         320 ケース
  引当済み:       -280 ケース（他拠点への確定発注 3件）
  ─────────────────────────
  利用可能在庫:    40 ケース  ← ここを見て発注量を決める
  入荷予定:       +200 ケース（4/6 工場からの配送）
  ─────────────────────────
  4/6以降の利用可能在庫: 240 ケース
```

- 利用可能在庫が0以下の場合は「現在補充不可（引当済み）」と赤字表示
- 入荷予定がある場合はその日付と数量を表示し、「いつなら補充可能か」の目安を提供

**受け入れ条件**:
- ATP値がリアルタイム（ページ表示時点）で計算されること
- ATP算出に含める発注ステータスを明示すること（confirmed / in_transit のみ。draft は含めない）
- ATP がマイナスの場合でも正しく表示されること（過剰引当の検知に使用）
- 入荷予定（配送 scheduled/in_transit）の加算は「予定」として視覚的に区別すること

**バックエンド実装方針**:
```python
def calc_atp(location_id: int, product_id: int, db: Session) -> dict:
    # 現在庫
    current = db.query(Inventory).filter(...).first().quantity

    # 確定済み出荷予定（引当済み）
    allocated = db.query(func.sum(Order.quantity)).filter(
        Order.from_location_id == location_id,
        Order.product_id == product_id,
        Order.status.in_(["confirmed", "in_transit"]),
        Order.actual_delivery_date.is_(None)  # 未着
    ).scalar() or 0

    # 入荷予定
    inbound = db.query(func.sum(DeliveryRecord.quantity)).filter(
        DeliveryRecord.to_location_id == location_id,
        DeliveryRecord.product_id == product_id,
        DeliveryRecord.status.in_(["scheduled", "in_transit"])
    ).scalar() or 0

    return {
        "current": current,
        "allocated": allocated,
        "atp": current - allocated,
        "inbound": inbound,
        "atp_with_inbound": current - allocated + inbound
    }
```
